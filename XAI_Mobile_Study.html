Lesson 1

왜 해석가능성이 필요한가?

ML 모델이 아무리 정확해도, "왜?"에 답할 수 없으면 현실에서 못 쓴다.

Machine Learning은 예측 성능이 뛰어나서 의사결정에 엄청난 잠재력이 있습니다. 하지만 XGBoost, SVM, Deep Neural Network 같은 모델들은 black box입니다. 인간이 이해하기엔 너무 복잡하죠.

문제는 단순히 "예측을 잘 맞추는 것"이 아니라 "왜 그렇게 예측했는지 이해하는 것"이 필요한 분야가 많다는 점입니다.

Black Box의 두 가지 문제

1. 신뢰 부족 (Hurts Trust) — "이 모델이 맞는다는 걸 어떻게 믿어요?"

2. 도입 장벽 (Creates Barriers) — 신뢰가 필요한 분야에서는 성능이 낮더라도 Linear Model 같은 전통적 방법을 선호

r/datascience
매니저: "이 모델 정확도 얼마야?"
DS: "97%입니다!"
매니저: "좋아. 근데 왜 이 고객이 이탈한다고 예측했어?"
DS: "..."
매니저: "설명 못 하면 못 써."
DS: *XAI 수업 수강신청*
GDPR & Legal

EU의 GDPR (General Data Protection Regulation)은 특정 상황에서 알고리즘 의사결정에 대한 "설명권(Right to Explanation)"을 요구합니다 (Goodman & Flaxman, 2017). 법적으로도 해석가능성은 선택이 아니라 의무가 되어가고 있습니다.

이해 확인

ML 모델의 해석가능성이 부족하면 생기는 두 가지 주요 문제는?

1. 신뢰 부족 (Hurts Trust) — 모델 결과를 믿을 수 없음.
2. 도입 장벽 (Creates Barriers) — 신뢰가 중요한 영역에서 ML 모델 채택이 어려움. 결과적으로 성능이 낮은 전통적 모델(예: linear model)을 대신 사용하게 됨.

Lesson 4

Interpretation Goals

"설명"이 필요한 이유는 하나가 아니라 네 가지다.

1. Discovery (발견)

데이터와 모델, 데이터 생성 과정에 대한 통찰을 얻기.
예: 자전거 대여 데이터에서 "연도(yr)"와 "온도(temp)"가 가장 중요한 Feature.

2. Debug & Audit (디버그)

모델이나 데이터의 결함을 발견하고 수정.
예: 탱크 인식 모델이 사실은 날씨만 봤던 사건!

3. Individual Predictions (개별 예측)

개별 의사결정을 이해하고 제어.
예: "대출이 왜 거절됐지? 어떻게 하면 승인돼?"

4. Fairness (공정성)

편향적이거나 차별적인 예측이 있는지 조사.
예: COMPAS 인종 편향 사건.

탱크 이야기 — Debug의 교과서적 사례

Cautionary Tale

전설적인 이야기입니다 (실제로 일어난 건 아니지만 교훈이 큼):

Neural Network로 사진에서 탱크를 인식하는 모델을 만들었습니다. 훈련 데이터에선 잘 됐는데, 실전에서 완전 실패. 왜? 탱크 사진은 흐린 날씨에, 탱크 없는 사진은 맑은 날씨에 찍었거든요. 모델이 학습한 건 탱크가 아니라 하늘이었습니다.

Thomas Dietterich 교수의 실제 경험: 곤충 54종을 사진으로 분류하는 프로젝트에서, 표본을 알코올에 넣고 한 상자씩 촬영했더니 날마다 기포 패턴이 달랐습니다. 모델은 곤충이 아니라 기포를 학습했죠. Saliency map으로 확인해서 발견.

r/learnmachinelearning
나: 정확도 99%의 곤충 분류 모델을 만들었다!

나: ...기포?

교훈: accuracy만 보면 바보가 된다. XAI로 "무엇을 학습했는지" 확인 필수.
이해 확인

Interpretation의 4가지 Goal을 나열하고, 각각 한 줄로 설명하세요.

1. Discovery — 데이터/모델/DGP에 대한 통찰 얻기 (어떤 Feature가 중요한지)
2. Debug & Audit — 모델/데이터의 결함을 찾아 수정 (탱크 이야기)
3. Individual Predictions — 개별 예측의 이유를 이해하고 제어 (대출 거절 사유)
4. Fairness — 편향/차별적 예측 조사 (COMPAS 사건)

해석가능성의 차원

Intrinsic, Model-specific, Model-agnostic — 해석 방법의 세 가지 유형

Intrinsic
모델 구조 자체가 단순해서 해석 가능
예: GLM, Decision Tree
Model-specific
특정 모델 구조를 활용
예: Gini Importance, LRP
Model-agnostic
어떤 모델이든 적용 가능 (Post-hoc)
예: SHAP, LIME, PFI
유형장점단점
Intrinsic추가 IML 불필요, 에러 원천 제거복잡한 관계 모델링 한계, Feature 많으면 어려움
Model-specific모델 구조를 활용해 효율적특정 모델에만 적용 가능
Model-agnostic어떤 모델이든 OK, 범용적모델 내부 구조 못 활용, Post-hoc이라 별도 단계 필요
r/datascience
Intrinsic: "저는 원래 투명인간입니다" (linear model)
Model-specific: "제 CT 사진을 보세요" (특정 병원만 가능)
Model-agnostic: "아무 모델이나 오세요, 다 설명해드립니다" (SHAP)

취업 면접 비유:
Intrinsic = 이력서가 한 장 (단순해서 바로 이해)
Model-agnostic = 어떤 지원자든 면접 질문으로 파악

Data Attribution — 머핀과 강아지

모델이 강아지 사진을 머핀으로 잘못 분류했습니다. Data Attribution으로 확인하면: "이 예측에 가장 큰 영향을 준 훈련 데이터"를 찾을 수 있습니다. 만약 그 훈련 데이터가 머핀처럼 생긴 사진들이라면? 모델이 왜 헷갈렸는지 이해가 됩니다.

Counterfactual — "뭘 바꾸면 되나요?"

대출 거절당한 사람에게: "당신의 기술 수준이 더 높고, 대출 금액을 $8,000으로 줄였다면 승인됐을 것입니다."

Counterfactual은 Feature 값의 최소한의 변경으로 원하는 결과를 얻는 방법을 알려줍니다. 가장 실용적인 설명 방식 중 하나.

r/learnmachinelearning
Feature Importance: "누가 범인이야?" (용의자 순위)
Feature Effects: "범인이 어떻게 했어?" (행동 방향)
Data Attribution: "목격자가 누구야?" (훈련 데이터)
Counterfactual: "어떻게 하면 범행을 막을 수 있었어?"

Local vs Global vs Regional

설명의 범위: 하나만 볼 건지, 전체를 볼 건지, 중간을 볼 건지

Local
개별 인스턴스 설명
예: LIME, SHAP, ICE
"이 한 명의 대출이 왜 거절?"
Global
전체 입력 공간 설명
예: PD plot, ALE, PFI
"모델 전체적으로 temp가 중요"
Regional
하위 집단/영역 설명
예: 근무일 vs 휴일 패턴
"근무일엔 출퇴근 피크"
자전거 대여 예시

Local (ICE 곡선): 특정 하루에 대한 시간-대여 관계 → 자세하지만 패턴 파악 어려움

Global (PDP): 전체 평균 곡선 → 깔끔하지만 이질성(heterogeneity) 숨김

Regional: 근무일 / 비근무일로 나눔 → 근무일은 아침&저녁 피크, 비근무일은 늦은 아침 피크. 디테일과 단순함의 균형!

r/datascience
Local: 이 환자 한 명을 MRI로 자세히 봄
Global: 전국민 건강 통계 평균
Regional: 20대 남성 vs 60대 여성으로 나눠서 봄

"평균의 함정"을 피하려면 Regional이 핵심
이해 확인

Global 설명의 한계는 무엇이고, Regional 설명이 이를 어떻게 해결하나요?

Global의 한계: Local 설명들을 평균내므로 이질성(heterogeneity)을 숨김. 서로 다른 패턴이 있어도 평균에 묻혀버림.
Regional의 해결: 데이터를 의미 있는 하위 집단으로 나눠서 각 집단의 패턴을 보존. 디테일을 잃지 않으면서도 과도한 복잡성을 피함.
Lesson 8

Correlation vs Dependence

상관관계가 0이라고 독립인 건 아니다 — 이 구분을 확실히 알아야 IML이 보인다.

Pearson Correlation (ρ)

ρ(X1, X2) = ∑(x1(i) - x̄1)(x2(i) - x̄2) / sqrt(∑(x1(i) - x̄1)² · ∑(x2(i) - x̄2)²)
• 범위: [-1, 1]. 오직 선형 관계만 측정!
• ρ > 0: 양의 선형 관계 (X1 오르면 X2도 오름)
• ρ < 0: 음의 선형 관계
• ρ = 0: 선형 상관 없음 (하지만 비선형 관계가 있을 수 있음!)

R² (Coefficient of Determination)

선형 모델 x̂2 = θ0 + θ1x1을 피팅한 후:

R² = 1 - SSELM / SSEconst

상수 모델(평균) 대비 선형 모델이 얼마나 잘 맞는지. 범위 [0, 1].

참고: ρ(X1, X2) = R (제곱근 관계)

Mutual Information (MI)

MI(X1, X2) = Ep(x1,x2)[log(p(x1,x2) / p(x1)p(x2))]
• Joint distribution이 marginal들의 곱과 얼마나 다른지 측정
• MI = 0 ⇔ 두 변수가 완전히 독립
• 선형/비선형 관계 모두 포착 가능! (Pearson과의 결정적 차이)
• 범주형 변수에도 적용 가능
핵심 차이

독립 ⇒ ρ = 0 (참)

ρ = 0 ⇒ 독립? 거짓! 비선형 의존성이 있을 수 있음.

예: 원형 분포 — ρ ≈ 0이지만 MI = 0.61 (강한 의존성)

유일한 예외: 두 변수가 jointly normal distribution이면 ρ = 0 ⇔ 독립

r/MachineLearning
Pearson: "두 분 사이에 선형 관계 없어요. 독립입니다."
MI: "잠깐, 원 모양으로 엮여있는데요?"
Pearson: "...선형 아니면 저는 못 봐요"

correlation != dependence
이걸 시험에 쓰면 교수님이 좋아합니다
이해 확인

Pearson ρ = 0인데 두 변수가 독립이 아닌 경우가 가능한가? 그렇다면 어떤 측도로 이를 탐지할 수 있나?

가능하다. Pearson은 선형 관계만 측정하므로, 비선형 의존성(예: 원형, U자형 관계)이 있으면 ρ = 0이어도 독립이 아닐 수 있음.
탐지 방법: Mutual Information (MI), Spearman correlation (rank 기반 단조 관계), HSIC (커널 기반) 등으로 비선형 의존성 감지 가능.
Lesson 9

Feature Interactions

Feature 하나의 효과가 다른 Feature 값에 따라 달라지면 — 그게 Interaction이다.

Dependence vs Interaction

Feature Dependence: 데이터 분포의 속성. "키가 큰 사람이 체중도 많이 나가는 경향" (X 사이의 관계)

Feature Interaction: 모델 또는 DGP 구조의 속성. "온도의 효과가 계절에 따라 달라짐" (X와 f(X) 사이의 관계)

의존성이 interaction을 유발할 수 있지만, 둘은 다른 개념입니다!

수학적 정의

E[(∂²f(x) / ∂xj∂xk)²] > 0
xj와 xk의 교차 편미분의 기댓값이 0보다 크면 interaction이 있다.
직관: xj가 f(x)에 미치는 영향이 xk 값에 따라 변하면 interaction.

Separable vs Non-separable

No Interaction (Separable)

f(x) = x1 + x2 + log(x1x2)

= x1 + log(x1) + x2 + log(x2)

= f1(x1) + f2(x2)

분리 가능! x1의 effect가 x2와 무관. 그래프에서 평행선으로 나타남.

Interaction (Non-separable)

f(x) = x1 + x2 + x1·x2

∂²f/∂x1∂x2 = 1 ≠ 0

분리 불가! x1의 기울기가 x2에 따라 변함. 그래프에서 기울기가 다른 선들로 나타남.

r/ProgrammerHumor
팀플 비유로 Interaction 이해하기:

No Interaction: A가 코딩, B가 디자인. 서로 독립적으로 일함.
A의 능력이 B 때문에 달라지지 않음. (평행선)

Interaction: A가 코딩, B가 PM. B가 잘하면 A의 생산성 UP,
B가 못하면 A도 삽질. A의 효과가 B에 의존! (기울기 변화)

separable function = 각자 따로 놀아도 되는 팀
non-separable = 시너지(or 갈등)가 있는 팀
이해 확인

f(x) = 3x1 + 2x2 + 5x1x2에서 x1과 x2 사이에 interaction이 있는가? 수학적으로 보이세요.

있다.
∂f/∂x1 = 3 + 5x2
∂²f/∂x1∂x2 = 5
E[(5)²] = 25 > 0
따라서 x1과 x2 사이에 interaction이 존재. x1이 f에 미치는 효과(기울기)가 x2 값에 따라 달라진다.

Lesson 1

왜 해석 가능한 모델을 쓰는가?

해석가능성을 얻는 가장 직접적인 방법: 애초에 해석 가능한 모델을 쓰는 것.

"해석 가능한 모델"이란 모델 구조 자체가 단순해서 사람이 직접 이해할 수 있는 모델입니다. Linear Model, Decision Tree, GAM 등이 여기에 해당하죠.

장점

본질적으로 해석 가능 — 별도 IML 방법 불필요 (에러 원천 제거)

학습 빠름 — 단순 구조 = 짧은 training time

Monotonicity — GLM 등은 Feature 커지면 결과도 일정 방향 (직관적)

친숙함 — 많은 사람이 이미 알고 있음 → 신뢰와 소통 용이

단점

가정 필요 — 가정이 틀리면 성능 저하

해석이 항상 쉽진 않음 — Feature 수백 개 + Interaction이면 Linear Model도 복잡

유연성 한계 — 복잡한 관계를 자동으로 모델링 못 함

모든 설명을 제공하지 않음 — Counterfactual 등은 여전히 별도 방법 필요

Interpretability vs Performance Trade-off

모델RMSER²해석가능성

권장 전략: 가장 단순한 모델에서 시작 → 성능이 부족하면 점진적으로 복잡도 증가. 가장 단순하면서 충분한 모델을 선택하라. (Cynthia Rudin: "항상 해석 가능한 모델을 먼저 시도해야 한다")

Lesson 2

Linear Regression

200년 넘게 쓰인 원조 해석 모델. 모든 IML의 출발점.

y = θ0 + θ1x1 + θ2x2 + ... + θpxp + ε
• y: 타겟 (예측하고 싶은 것)
• θj: Feature xj의 가중치 (이게 해석의 핵심!)
• θ0: 절편 (모든 Feature가 0일 때의 기대값)
• ε: 잔차 (모델이 설명 못 하는 노이즈)

가정 (Assumptions)

1. Feature와 타겟 사이에 선형 관계

2. 잔차 ε이 정규분포, 등분산 (homoscedastic): ε ~ N(0, σ²)

3. 관측치 간 독립 (반복 측정 없음)

4. Feature와 잔차가 독립

5. 다중공선성(Multicollinearity) 없거나 적음

가정 위반해도 모델은 돌아감! 하지만 p-value 등 inference 기반 지표는 신뢰 불가.

Weight 해석법

수치형 Feature

xj가 1단위 증가하면 y가 θj만큼 변화, ceteris paribus (다른 조건 동일)

예: 온도 1도 오르면 자전거 대여 120.5건 증가

범주형 Feature

Dummy encoding: L개 범주 → L-1개 이진 Feature. 빠진 하나가 reference category.

예: 봄은 겨울(ref) 대비 862건 더 대여

Feature Importance: t-statistic

|tθj| = |θ̂j| / SE(θ̂j)
• θ̂j를 표준오차(SE)로 나눈 것. 추정의 신뢰도 대비 크기를 나타냄.
• |t| 클수록 = 중요한(significant) Feature
• p-value: H0: θj = 0 하에서 이 t값 이상이 나올 확률. 작을수록 "0이 아니다"에 강한 근거.
Bike Data 예시

θ0 (Intercept)      = 3229.3  (겨울, 모든값 0일때)
seasonSPRING        = +862.0  (p=0.00, 유의함)
seasonSUMMER        = +41.6   (p=0.81, 유의하지 않음!)
temp                = +120.5  (1도당 120건 증가)
hum                 = -31.1   (습도 높으면 감소)
days_since_2011     = +4.9    (t=26.9, 가장 중요!)

Linear Regression 해석법 요약:

수치형: "1 올리면 결과가 theta만큼 변해요" (ceteris paribus)
범주형: "이 그룹은 reference 대비 theta만큼 달라요"
절편: "다 0이면 이만큼이에요"

주의: "ceteris paribus"를 까먹으면 해석이 틀려진다
인과관계 아님! 상관관계를 과잉 해석하지 말 것!
이해 확인

자전거 대여 데이터에서 seasonSUMMER의 weight가 41.6이고 p-value가 0.81이면, 이를 어떻게 해석해야 하나요?

weight 41.6은 겨울(reference) 대비 여름의 대여량이 41.6건 더 많다는 의미지만, p-value가 0.81로 매우 크므로 θj = 0이라는 귀무가설을 기각할 수 없음.
즉, 이 차이는 통계적으로 유의하지 않음. 여름과 겨울의 차이가 노이즈 수준이라는 뜻. (다른 Feature들로 이미 설명되고 있을 수 있음)
Lesson 3

Interactions, High-order & LASSO

Linear Model을 더 유연하게, 그리고 다시 간결하게.

Interaction & High-order Effects

기본 LM에 교차항이나 고차항을 추가하면 더 복잡한 관계를 모델링할 수 있습니다.

y = θ0 + θ1x1 + θ2x2 + θ12x1x2 + θx²x1² + ε
• θ12x1x2: Interaction term (두 Feature의 곱)
• θx²x1²: Quadratic (고차) term
• 유연성 증가, 하지만 해석이 복잡해짐. 수동 지정 필요.
Interaction 해석 예시 (Bike Data)

temp × season interaction을 추가하면:

겨울: 1도 상승 → +160.5건 (reference)
봄:   1도 상승 → +109.8건 (160.5 - 50.7)
여름: 1도 상승 → -61.5건  (160.5 - 222.0) !!
가을: 1도 상승 → +187.7건 (160.5 + 27.2)

여름에는 온도가 오르면 오히려 대여가 줄어든다! (너무 더워서) 이게 Interaction의 힘.

모델R²adj. R²
Simple LM (Main only)0.850.84
High-order effects0.870.87
Interaction effects0.960.93

LASSO — 불필요한 Feature 자동 제거

minθ (1/n) ∑(y(i) - x(i)Tθ)² + λ||θ||1
• L1 정규화: |θ| 합에 페널티 부여
• 효과: 일부 weight를 정확히 0으로 만듦 → Feature Selection!
• λ가 클수록 더 많은 Feature가 탈락. CV로 최적 λ 선택.
r/ProgrammerHumor
LASSO는 Feature들의 생존 오디션

lambda = 0: "여러분 다 합격입니다" (과적합)
lambda = 적당: "핵심 인력만 남겨주세요"
lambda = 너무 큼: "전원 해고" (weight 다 0)

LASSO = ML 세계의 구조조정
주의: Group LASSO

일반 LASSO는 범주형 변수의 개별 dummy를 따로 처리합니다. 예를 들어 season의 4개 level 중 2개만 0이 될 수 있음. 범주형 Feature를 통째로 선택/제거하려면 Group LASSO (Yuan & Lin, 2006)를 사용해야 합니다.

Lesson 4

GLM & Logistic Regression

타겟이 연속형이 아니면? Link function으로 확장한다.

GLM (Generalized Linear Model)

g(E(y|x)) = xTθ  ⇔  E(y|x) = g-1(xTθ)
• g: Link function — 선형 예측자를 타겟의 기대값에 연결
• LM은 GLM의 특수 사례: 정규분포 + identity link (g(x) = x)
• 다른 분포(Bernoulli, Poisson 등)와 link function을 지정하면 다양한 타겟 처리 가능

Logistic Regression — 분류의 해석 모델

Setup

Bernoulli 분포 + logit link function:

g(x) = log(x / (1-x))    (logit)
g-1(x) = 1 / (1 + exp(-x))  (sigmoid)

결과: π(x) = P(y=1|x) = 1 / (1 + exp(-xTθ))

보통 π(x) > 0.5이면 class 1, 아니면 class 0으로 예측.

Weight 해석 — Odds Ratio

1
Log-odds 해석 (직관적이지 않음)
xj가 1 증가 → log-odds가 θj만큼 변화
2
Odds Ratio 해석 (더 직관적!)
xj가 1 증가 → odds가 exp(θj) 배로 변화
Bike Data 이진 분류 예시

타겟: "높은 대여량" (cnt > 5531) vs "낮은/중간 대여량"

temp의 weight = 0.29 → exp(0.29) = 1.34

해석: 온도 1도 상승 → "높은 대여량"의 odds가 1.34배 증가, c.p.

r/datascience
Logistic Regression 해석의 3단계:

1. weight 봄: "0.29? 그래서 뭐?"
2. exp() 씌움: "1.34배? 오 좀 이해되네"
3. 면접관에게 설명: "온도 1도 올리면 확률이 1.34배요"
면접관: "odds랑 probability 차이는?"

...odds = p/(1-p) 입니다 (확률과 다름!)
이해 확인

Logistic Regression에서 Feature의 weight가 -0.5일 때, odds ratio는 얼마이고 어떻게 해석하나요?

Odds ratio = exp(-0.5) = 0.607
해석: 해당 Feature가 1단위 증가하면, class 1의 odds가 약 0.61배로 감소 (약 39% 감소), 다른 조건 동일.
쉽게 말해: 이 Feature가 올라가면 class 1이 될 가능성이 줄어든다.
Lesson 5

GAM — 비선형이지만 해석 가능

Linear Model의 직선이 답답할 때, 곡선으로 업그레이드하되 해석은 유지.

g(E(y|x)) = θ0 + f1(x1) + f2(x2) + ... + fp(xp)
• fj(xj): 각 Feature에 대한 유연한 함수 (예: spline)
• θjxj 대신 fj(xj)를 쓰므로 비선형 효과 모델링 가능
• 하지만 additive 구조는 유지 → 각 Feature의 효과를 개별적으로 시각화 가능
LM과의 비교

LM: 온도와 대여량이 직선 관계 (기울기 하나로 설명)

GAM: 온도 20도까지는 급증, 그 이후로는 완만해지는 곡선 관계 (spline으로 포착)

더 유연하고 정확하지만, weight 하나로 요약이 안 됨 → 시각적 해석이 필요.

해석 방법

Feature Effect Plot

각 fj(xj)를 그래프로 시각화. 평균 예측 대비 각 Feature 값에서의 효과를 봄.

EDF (Effective Degrees of Freedom)

smoothness의 복잡도. edf = 1이면 직선(LM과 동일), 클수록 복잡한 곡선.

LM에서 비선형 처리 우회법

GAM 없이 LM에서 비선형을 다루는 방법들:

1. Feature 변환 (log, exp 등)

2. 고차항 추가 (x², x³)

3. Feature를 구간으로 나누기 (binning)

하지만 이런 우회법들은 수동이고, GAM은 자동으로 smooth한 비선형 관계를 학습합니다.

r/MachineLearning
LM: y = ax + b (세상이 직선이면 좋겠다)
GAM: y = f(x) (곡선이지만 한 변수씩 따로 본다)
Neural Net: y = f(x1, x2, ..., xp) (모든 게 얽혀있음)

GAM은 LM과 Neural Net 사이의 sweet spot
"곡선은 그리되, 각자 따로 그려주세요"
Lesson 6

Model-based Boosting

Boosting인데 해석 가능? 단순한 Base Learner를 쓰면 가능하다.

핵심 아이디어

일반적인 Gradient Boosting은 복잡한 트리를 쌓아서 해석이 어렵습니다. 하지만 단순한 base learner (단일 Feature의 linear model 또는 spline)를 사용하면?

각 iteration에서 가장 좋은 BL을 선택하고 추가. 같은 Feature의 BL들은 합쳐짐. 결과: GAM과 동일한 additive 구조!

f̂ = f̂0 + f̂1(x1) + f̂3(x3) + ...
각 component function은 해석 가능. Early stopping으로 regularization + feature selection 효과까지!

해석 도구

Feature Importance

각 iteration에서 선택된 BL이 risk(MSE)를 얼마나 줄였는지 Feature별로 합산.

예: days_since_2011이 첫 iteration에서 risk를 140,783 줄임 → 가장 중요!

Feature Effect

각 Feature의 component function fj(xj)를 시각화. Linear BL과 spline BL의 효과를 합친 것.

r/learnmachinelearning
일반 Boosting: 깊은 트리 수백 개 쌓기 (black box)
Model-based Boosting: 단순 모델 수백 개 쌓기 (투명 box)

같은 boosting인데 base learner 하나로
"해석 가능"과 "해석 불가능"이 갈린다

결론: 복잡한 건 base learner지, boosting이 아니었다
Lesson 7

Decision Trees

if-else의 연속. 사람이 가장 직관적으로 이해할 수 있는 모델.

f̂(x) = ∑m=1M cm · 1{x ∈ Rm}
• 데이터를 Feature의 cut-off 값으로 분할 (greedy search)
• 각 leaf node Rm에서 상수 평균 cm을 예측
• 회귀(regression)와 분류(classification) 모두 가능
Decision Tree의 강점

1. Interaction과 비선형 효과를 자동으로 모델링

2. 혼합 Feature space (수치+범주) 및 결측치 처리 가능

3. 트리 구조를 따라가면 바로 해석 가능

해석 방법

트리 따라가기

루트 → 리프까지 decision rule의 연속.
예: "2011년 이후 105일 미만 → 평균 대여 1,798건 (데이터의 15%)"

Feature Importance

각 Feature가 사용된 모든 split에서의 "split criterion 개선"을 합산.
예: days_since_2011 = 79.53, temp = 17.55

CART의 문제점

Selection Bias

문제: CART는 cardinality가 높은(값이 많은) Feature를 편향적으로 선택합니다. 연속형 변수가 이진 변수보다 split 후보가 많으니까요.

해결: Conditional Inference Tree (ctree), Model-based Recursive Partitioning (mob)

ctree/mob은 Feature 선택과 split point 결정을 분리하고, hypothesis test를 stopping criteria로 사용합니다.

AND 요일 != 수요일 AND 풍속 > 12.1 AND..."

...이쯤 되면 Linear Model보다 해석이 어렵다
tree depth는 해석가능성의 적

CART가 연속형 변수를 범주형 변수보다 선호하는 이유는? 이를 해결하는 방법은?

Lesson 3

Partial Dependence Plot (PDP)

Friedman (2001). ICE 곡선들의 평균 = 전체적인 Feature 효과

One-Liner

PDP = ICE 곡선들을 세로축 방향으로 평균 낸 것.

학생 30명의 중간고사 성적 변화 곡선(ICE)을 평균 내면 반 전체 평균 성적 변화 추이(PDP)가 됩니다.

fS,PD(xS) = Ex-S[f̂(xS, x-S)] ≈ 1/n ∑i=1n f̂(x*S, x(i)-S)
• 관심 Feature xS를 특정 값으로 고정
• 나머지 Feature x-S의 marginal 분포에 대해 기대값 계산
• 실제로는 모든 관측값의 ICE 값을 grid point별로 평균

선형 회귀에서의 PDP

Example

f̂(x1, x2) = β1x1 + β2x2 + β0 일 때,

f1,PD(x1) = β1x1 + β2·E[x2] + β0 = β1x1 + const

PDP가 회귀 계수의 효과를 그대로 보여줌! 선형 모델에서는 깔끔하게 동작.

PDP 해석하기 — 자전거 대여 예시

Bike Sharing

• 온도가 올라갈수록 자전거 대여 예측값 증가
• 약 25도를 넘으면 다시 소폭 감소 (너무 더워서)
• ICE 곡선들이 대체로 평행 = 온도 효과는 다른 Feature와 interaction 적음

Centered ICE (C-ICE)

ICE 곡선들이 세로 방향으로 흩어져 있으면 패턴 파악이 어렵습니다. 해결책: 모든 곡선을 한 지점(보통 min 값)에서 0으로 맞추기.

f̂(i)S,cICE(xS) = f̂(i)S(xS) - f̂(i)S(x')
x'은 기준 값 (보통 xS의 최솟값).
해석: "기준점 대비 예측이 얼마나 변했는가"
예: "습도 0% 대비 97%일 때 자전거 대여가 약 1000대 감소"

코드 리뷰에서도 전체 코드보다 diff가 유용한 것과 같은 이치

Lesson 4

Extrapolation과 Interaction 문제

PDP의 치명적 약점: 상관 Feature가 있으면 가짜 데이터로 예측함

Extrapolation 문제 (핵심!)

The Problem

PDP는 관심 Feature xS를 고정하고 나머지 x-S를 marginal 분포에서 샘플링합니다.

문제: xS와 x-S가 상관되어 있으면, 이 조합이 현실에 존재하지 않는 데이터를 만듭니다.

Example

"키"와 "몸무게"가 강하게 상관된 데이터에서 PDP를 구한다고 합시다.

PDP(키=190cm)를 계산할 때, 몸무게는 전체 분포에서 가져옵니다.
→ 키=190cm + 몸무게=40kg 같은 비현실적 조합이 만들어짐!

모델이 이런 조합을 본 적이 없으니 이상한 예측을 하고,
그 이상한 예측이 PDP 평균에 섞여 들어가면서 결과가 왜곡됩니다.

r/MachineLearning
PDP의 extrapolation:
"연봉 2000만원인데 강남 아파트 살고 있는" 가상 인간을 만들어서
모델한테 "이 사람 대출 상환율 얼마야?" 라고 물어보는 격

모델: "그런 사람 본 적 없는데요..." *아무 숫자나 내뱉음*

Interaction 문제

Averaging hides interactions

PDP는 ICE 곡선들을 평균 내니까, 개별 곡선의 다양한 모양이 사라집니다.

예: 여름에는 온도 효과가 크고 겨울에는 작은데, 평균 내면 이 차이가 뭉개집니다.

해결: ICE + PDP를 함께 그려야 합니다. ICE 모양이 다르면 interaction 존재!

2D PDP로 Interaction 확인

Bike Sharing Example

온도 × 습도 2D PDP를 그리면:

• 낮은~중간 습도 + 높은 온도 = 자전거 많이 탐

• 높은 값에서 ICE 곡선 모양이 달라짐 = interaction 있음

Check

PDP의 extrapolation 문제가 발생하는 근본 원인은?

PDP가 x-S의 marginal 분포 P(x-S)에서 샘플링하기 때문.
Feature들이 상관되어 있으면 P(xS, x-S) ≠ P(xS)P(x-S)인데,
PDP는 마치 독립인 것처럼 marginal에서 가져오므로 joint distribution 밖의 데이터가 생김.
Lesson 5

ALE Plot — Accumulated Local Effects

상관 Feature가 있어도 정확한 Feature 효과를 보여주는 방법

ALE가 해결하는 두 가지 문제

PDP의 문제: marginal 분포 사용 → extrapolation

M Plot의 문제: conditional 분포 사용 → omitted variable bias (OVB)

ALE의 해결: conditional 분포 + 편미분 → 두 문제 모두 해결!

왜 M Plot은 안 되는가?

M Plot (Marginal Plot)

M Plot은 x-S를 conditional distribution P(x-S|xS)에서 가져옵니다. Extrapolation은 해결되지만...

OVB 문제: x1과 x2가 상관되어 있으면, M Plot(x1)에 x2의 효과도 섞여 들어옵니다.

예: y = -x1 + 2x2, Cor(x1,x2)=0.9면 → M Plot(x1)이 양의 기울기를 보임. 실제 x1 효과는 음수인데!

ALE의 핵심 아이디어: 편미분 + 적분

1
Local Effects (편미분)
∂f̂/∂xS를 구함 → 다른 Feature의 main effect가 제거됨!
↓
2
Conditional 평균
P(x-S|xS)에 대해 평균 → extrapolation 방지!
↓
3
Accumulate (적분)
min(xS)부터 x까지 누적 → 원래 효과 복원!
↓
4
Centering
평균을 빼서 E[fALE] = 0으로 만듦
Why 편미분이 핵심인가

f̂(x1, x2) = 2x1 + 2x2 - 4x1x2 일 때:

∂f̂/∂x1 = 2 - 4x2 ← x2의 main effect(2x2)가 사라졌다!

이걸 다시 x1에 대해 적분하면: [2x1 - 4x1x2] ← x1의 효과 + interaction만 남음

OVB가 원천적으로 차단됩니다.

실제 계산: Finite Differences로 근사

f̂S,ALE(x) = ∑k=1kS(x) 1/nS(k) ∑i:x(i)S∈[zk-1,zk] [f̂(zk, x(i)-S) - f̂(zk-1, x(i)-S)]
• Feature 범위를 K개 구간(interval)으로 나눔 (보통 분위수 사용)
• 각 구간 내 관측값들에 대해 구간 경계에서의 예측 차이(finite difference) 계산
• 구간별 평균 = local effect
• local effect를 누적(accumulate) → ALE

M Plot: "주변 데이터만 쓰자" (OVB 함정)
ALE: "변화율만 보고 누적하자" (elegant한 해결)

미적분을 이렇게 쓰는 거 보면 수학과 교수님들 뿌듯하시겠다
Check

ALE에서 "Accumulated"의 의미는?

Local effects(편미분 근사)를 구간별로 누적(적분)한다는 뜻입니다.
편미분으로 다른 Feature의 main effect를 제거하고,
다시 적분해서 원래 Feature의 effect를 복원합니다.

Difference 1: ALE는 예측값 자체가 아니라 예측의 변화(partial derivative)를 conditional distribution에서 평균. → extrapolation 방지!

  • Feature들이 상관되어 있을 때
  • 모델에 interaction이 있을 때
  • Feature의 "순수한" main effect를 보고 싶을 때
PDP가 괜찮을 때
  • Feature들이 (거의) 독립일 때
  • ICE 곡선과 함께 보고 싶을 때
  • 해석이 더 직관적 (y축 = 예측값)

Lesson 2

Shapley Value — 공정 분배의 수학

모든 가능한 순서를 고려하는, 수학적으로 유일하게 공정한 방법

계산 3단계

1
모든 입장 순서를 나열
3명이면 3! = 6가지: (A,B,C), (A,C,B), (B,A,C), (B,C,A), (C,A,B), (C,B,A)
2
각 순서에서 한계 기여도 계산
j가 들어오기 전 vs 후의 가치 차이 = v(S∪{j}) - v(S)
3
전체 순서의 한계 기여도를 평균
이 평균 = Shapley value φj

공식 (Set Definition)

φj = ∑S⊆P\{j} |S|!(|P|-|S|-1)!/|P|! · [v(S∪{j}) - v(S)]
• v(S∪{j}) - v(S) : j 합류 시 가치 증가분
• |S|!(|P|-|S|-1)!/|P|! : 가중치. 해당 크기의 팀이 만들어질 확률
• 모든 조합에 대해 가중 평균

공식 (Order Definition) — 더 직관적

φj = 1/|P|! · ∑τ∈Π [v(Sτj∪{j}) - v(Sτj)]
모든 순서(τ)에서 j 앞에 있는 사람들 = Sτj.
j가 합류했을 때 가치 증가를 구하고, 전체 평균!

예: 순서 (B, A, C)에서 A 앞에 있는 사람 = {B}
→ A의 기여 = v({B,A}) - v({B})

두 정의는 수학적으로 완전히 동일합니다. Order 정의의 장점: 랜덤 샘플링으로 근사 가능!
r/math
Set definition: O(2n) combinations 계산해야 함
Order definition: O(n!) permutations 계산해야 함
둘 다 exponential이지만 Order는 샘플링이 된다는 게 포인트
"같은 NP-hard인데 하나는 approximate가 되네ㅋㅋ"

ML에 적용하기

게임 이론의 Player를 Feature로, 게임을 예측으로 바꾸면 된다

Game Theory→Machine Learning

Value Function

v(S) = f̂S(xS) - E[f̂(x)]
• f̂S(xS) : Feature 집합 S만 사용한 예측 (나머지는 평균 대체)
• E[f̂(x)] : 전체 데이터 평균 예측
• 빼기를 해서 v(∅) = 0을 보장 (Shapley 필수 조건)

Marginalization

ML 모델은 모든 Feature를 입력받아야 합니다. 일부만 쓰려면?

해결: 안 쓰는 Feature를 실제 데이터에서 랜덤으로 뽑은 값으로 대체하고, 여러 번 예측 후 평균.

근사 알고리즘

정확한 계산은 NP-hard. 현실적으로는 샘플링으로 근사한다.

Feature 10개 → 10! = 3,628,800 순서
Feature 20개 → 20! ≈ 2.4 × 1018 순서

정확한 계산은 현실적으로 불가능합니다.

해결: M개의 랜덤 순서만 샘플링

φj ≈ 1/M ∑m=1M [f̂(x(m)+j) - f̂(x(m)-j)]
M번 반복해서 한계 기여도의 평균. M이 클수록 정확하지만 느려짐.

알고리즘 (Strumbelj et al., 2014)

1
랜덤 순서 τ 뽑기
↓
2
j 앞의 Feature 집합 Sm 결정
↓
3

↓

인공 관측값 2개 생성
x+j: Sm∪{j}는 x에서, 나머지는 z에서

Kernel SHAP — 5단계

Model-agnostic SHAP 계산법. 어떤 모델이든 OK.

Step 1 · Coalition 샘플링

z' ∈ {0,1}p 형태로 Feature 조합 생성.

Step 2 · Feature Space 변환 & 예측

1인 Feature는 x에서, 0인 Feature는 랜덤 데이터에서 가져와 예측.

Step 3 · Kernel 가중치 (핵심)
π(z') = (p-1) / [C(p,|z'|) · |z'| · (p-|z'|)]
직관: Feature 1~2개만 있는 coalition이나 거의 다 있는 coalition에 높은 가중치.
왜? 개별 Feature의 "순수한" 효과를 격리해서 볼 수 있으니까.
∅과 전체 집합은 가중치 ∞ → 제약조건으로 처리.
Step 4 · 가중 선형 회귀 (WLS)

g(z') = φ0 + ∑φjz'j
φ0 = E[f̂] 고정. 나머지는 WLS로 추정.

Step 5 · SHAP values 반환

추정된 φj = SHAP values.
4515 + 34 - 1654 - 323 = 2572 ≈ 2573

PFI — Permutation Feature Importance

Feature를 셔플해서 "파괴"하고, 성능이 얼마나 떨어지는지 측정

Core Idea

면접에서 지원자의 "학력" 칸을 랜덤으로 섞어버리면 어떻게 될까요?

만약 HR 담당자(모델)가 학력을 중요하게 봤다면, 합격 예측 정확도가 크게 떨어질 겁니다.

반대로 학력을 안 봤다면? 셔플해도 예측 정확도가 그대로!

PFI = "Feature를 셔플 → 성능 변화 측정"

PFI 계산 과정

1
원본 성능 측정
원본 데이터 D로 Remp(f̂, D)를 계산
↓
2
Feature xj를 랜덤 셔플 (permute)
xj의 marginal 분포 P(xj)는 유지하되, y 및 다른 Feature와의 관계를 파괴
↓
3
셔플된 데이터로 성능 측정
Remp(f̂, D̃S)를 계산
↓
4
차이 = PFI
여러 번 반복 후 평균
PFIS = 1/m ∑k=1m [Remp(f̂, D̃S(k)) - Remp(f̂, D)]
• Remp : 경험적 리스크 (예: MAE, MSE)
• D̃S(k) : Feature xS가 k번째 셔플된 데이터
• m : 반복 횟수 (랜덤성 줄이기 위해)
• PFI가 크면 = 해당 Feature가 중요!
Bike Sharing 예시

yr(연도)과 temp(온도)가 가장 중요한 Feature.
yr을 셔플하면 MAE가 816 증가 → 연도 정보가 파괴되면 성능이 크게 떨어짐.
error bar = 여러 번 셔플한 결과의 5%/95% 분위수.

EXAM ALERT

Final Exam Task 4, 5의 핵심 논점입니다. 이 섹션을 3번 읽으세요.

문제 1: Extrapolation

The Setup

y = x3 + ε 에서 x1과 x2가 매우 강하게 상관 (x2 = x1 + ε2, ε2 아주 작음).
선형 모델을 fit하면: f̂(x) ≈ 0.3x1 - 0.3x2 + x3

핵심: 실제 데이터에서 0.3x1 - 0.3x2 ≈ 0 (서로 상쇄).
즉, x1과 x2는 실질적으로 예측에 무관합니다.

PFI가 틀리는 이유

x1을 셔플하면 x1과 x2의 상관 관계가 깨짐.
→ 0.3x1 - 0.3x2 ≠ 0이 되는 비현실적 조합 발생!
→ 모델이 이런 조합에서 이상한 예측 → 성능 하락
→ PFI > 0 → "x1이 중요하다"라는 잘못된 결론!

x1, x2가 실제로는 예측에 무관한데 PFI는 높게 나옴.

r/MachineLearning
PFI의 extrapolation 문제를 비유하면:

"쌍둥이 직원 A, B가 있는데 항상 같이 일함.
A를 랜덤 부서로 보내면 A 없이도 일이 되는 게 아니라
A가 엉뚱한 곳에서 사고를 쳐서 전체 성과가 떨어짐.
결론: A가 중요해서가 아니라 A를 잘못된 곳에 배치해서 망한 것"

문제 2: Interaction 포함

Interaction Example

y = x1x2 + x3 + ε (x1, ..., x4는 독립)

x3 혼자서 x1·x2만큼 기여하는데, PFI 결과는 x1 = x2 = x3 (동일!)

왜? x1을 셔플하면 x1x2 interaction까지 파괴됨.
즉, PFI(x1)에 interaction 효과가 통째로 포함됨.
x1의 "개별 기여"와 "interaction 기여"가 구분 안 됨.

문제 3: Training vs Test Data

Overfitting Detection

독립적인 20개 Feature로 y를 예측 (실제로 관계 없음). XGBoost가 50개 데이터로 overfitting.

Train PFI: 일부 Feature가 중요해 보임 (spurious correlation 잡음)
Test PFI: 모든 Feature가 중요도 0 (올바른 결과)

→ 일반화 성능에 관심 있으면 반드시 test data에서 PFI!

Check

x1과 x2가 강하게 상관되어 있을 때, PFI(x1)이 높게 나오면 x1이 정말 중요한 건가요?

반드시 그런 것은 아닙니다.

PFI(x1)이 높은 이유가 두 가지일 수 있음:
1) x1이 실제로 중요함
2) x1을 셔플해서 x1-x2 상관이 깨지면서 비현실적 데이터가 생겨 성능이 떨어진 것 (extrapolation)

구분하려면 CFI와 비교하거나 PIMP로 유의성 검정 필요.

PIMP — Testing Importance

Altmann et al. (2010). PFI 결과가 "진짜 중요한 건지" 통계적으로 검증

Why PIMP?

PFI는 랜덤 셔플을 하기 때문에 값이 0이 아닐 수도 있습니다 (stochasticity). PIMP는 "이 PFI가 우연히 나온 건 아닌지" 검정합니다.

PIMP 알고리즘

1
H0 생성: target y를 셔플
y를 셔플하면 모든 Feature와 y의 관계가 끊김 (H0: Feature 무관)
↓
2
셔플된 y로 모델 재학습 & PFI 계산
이 PFI = H0 하에서의 PFI 분포 (nrep번 반복)
↓
3
원래 y로 PFI 계산 (H1)
H0 분포와 비교하여 p-value 산출
Extrapolation Example에 PIMP 적용

x1, x2는 상관되어 있지만 y와 무관한 예시에서:
• PFI(x1), PFI(x2) > 0 (extrapolation 때문에 높게 나옴)
• PIMP p-value > 0.05 → "통계적으로 유의하지 않음"
• PIMP가 PFI의 거짓 양성을 걸러냄!

Multiple Testing

Feature가 많으면 테스트도 많아짐 → Bonferroni correction: α/m 으로 유의 수준 보정. 보수적이지만 안전합니다.

PFI: "이 Feature 중요해요!"
PIMP: "근거 있어? p-value 보여줘"
PFI: "..."
PIMP: "p=0.3이네. 유의하지 않음. 기각."

통계학의 힘: 직감이 아닌 데이터로 판단
Lesson 5

CFI — Conditional Feature Importance

Strobl et al. (2008). PFI의 extrapolation 문제를 해결하는 "조건부 셔플"

PFI vs CFI 핵심 차이

PFI: xj를 marginal 분포 P(xj)에서 셔플 → 다른 Feature와의 관계 파괴 → extrapolation

CFI: xj를 conditional 분포 P(xj|x-j)에서 셔플 → joint distribution 보존 → extrapolation 없음!

CFIS = 1/m ∑k=1m [Remp(f̂, D̃S|-S(k)) - Remp(f̂, D)]
PFI와 공식 구조는 같지만, 셔플 방식이 다름.
• PFI: xS를 전체 marginal에서 셔플
• CFI: xS를 x-S에 조건부로 셔플 (예: 비슷한 x-S 값을 가진 그룹 내에서만 셔플)

CFI가 측정하는 것: "고유 기여 (Unique Contribution)"

CFI가 파괴하는 건 xj의 고유한 정보만입니다.
xj가 x-j와 공유하는 정보는 conditional 셔플에서 보존되므로 파괴되지 않음.

• xS ⊥ y | x-S (조건부 독립) → CFI = 0 (고유 정보 없음)
• 모델이 xS를 안 쓰면 → CFI = 0

Extrapolation Example 재방문

y = x3 + ε, x1과 x2 강하게 상관, f̂(x) ≈ 0.3x1 - 0.3x2 + x3:

PFI: x1, x2 모두 중요하다고 나옴 (extrapolation 때문)
CFI: x1, x2 모두 중요하지 않다고 나옴!

왜? x1을 x2에 조건부로 셔플하면 x1 ≈ x2 관계가 유지됨
→ 0.3x1 - 0.3x2 ≈ 0이 여전히 성립 → 성능 변화 없음 → CFI ≈ 0

CFI의 해석 목표 답변

해석 목표CFI 답변
(1) Causal for prediction?CFI ≠ 0 → 모델이 xj 사용
(2) Prediction-relevant info?xj ⊥ y | x-j이면 CFI = 0 (공유 정보는 감지 안 함)
(3) Unique contribution?CFI ≠ 0 → xj가 고유 정보 제공 (최적 답변!)
r/datascience
PFI: "이 직원을 랜덤 부서로 보내보자" (다른 부서에서 사고남)
CFI: "이 직원을 비슷한 역할을 하는 다른 사람으로 교체해보자"

CFI가 더 공정한 평가인 건 자명한데,
"비슷한 역할의 다른 사람"을 찾는 게 어렵다는 게 문제
(conditional distribution 추정이 어려움)
Check

x1과 x2가 상관되어 있고 둘 다 y와 관련 있을 때, CFI(x1) = 0이 될 수 있나요?

네! x1 ⊥ y | x-1(= x2 포함)이면 CFI(x1) = 0.
즉, x1이 y와 관련이 있더라도 그 정보가 x2를 통해 이미 알 수 있다면,
x1의 고유 기여는 0.
"x2가 x1을 대신할 수 있다"는 뜻.
Lesson 6

LOCO — Leave One Covariate Out

Feature를 아예 빼고 모델을 다시 학습시킨 뒤 성능 비교

Core Idea

PFI/CFI와 근본적으로 다릅니다:

• PFI/CFI: 같은 모델에서 Feature를 "셔플"

• LOCO: Feature를 제거하고 모델을 다시 학습

LOCO는 "이 모델"이 아니라 "이 학습 알고리즘"의 Feature 의존도를 측정

LOCOj = Remp(f̂-j) - Remp(f̂)
• f̂: 모든 Feature로 학습한 모델
• f̂-j: Feature j를 빼고 재학습한 모델
• LOCOj > 0: Feature j 없이는 성능이 떨어짐

LOCO의 한계

Correlated Features Example

x1과 x2가 강하게 상관, y = x2 + x3 + ε

Local Explanation이 필요한 이유

전체 모델을 이해하려 하지 말고, "이 한 건"만 설명하면 된다.

The Situation

은행 대출 심사 AI가 당신의 신청을 거절했습니다. 이유는 안 알려줍니다.

당신이 듣고 싶은 건 "이 모델의 전체 작동 원리"가 아니라, "내 신청이 왜 거절됐는지"입니다.

이것이 바로 Local Explanation의 핵심 동기입니다.

r/learnmachinelearning
Global explanation: "이 모델은 100만 개의 가중치를 사용해서..."
사용자: "...그래서 내 대출은 왜 거절됐는데요?"

Local explanation: "당신의 소득 대비 대출 금액이 높아서입니다"
사용자: "아, 그러면 금액을 줄여볼게요"

Lesson 2

LIME 핵심 아이디어

복잡한 모델이라도, 한 점 주변만 보면 단순한 모델로 설명할 수 있다.

Core Idea

LIME = Local Interpretable Model-agnostic Explanations

아무리 복잡한 ML 모델이라도, 특정 데이터 포인트 근처에서는 간단한 모델(선형 회귀, 결정 트리)로 근사할 수 있다는 아이디어.

지구는 둥글지만, 내 집 앞 도로는 평평합니다. LIME은 이 원리를 ML에 적용한 것입니다.

Surrogate Model이란?

surrogate = "대리인". 블랙박스 모델을 대신해서 설명해주는 간단한 모델입니다.

선형 모델 (가장 흔함)

결정 트리

2. 답변 패턴을 보고 간단한 규칙을 만든다

LIME에서 "Model-agnostic"이 의미하는 것은?

블랙박스 모델이 어떤 종류든 상관없다는 뜻입니다. Random Forest든, Neural Network든, SVM이든 — LIME은 모델 내부를 볼 필요 없이 입력과 출력만 사용합니다. 모델을 "f(x) = ?"로만 취급하기 때문에 어떤 모델에든 적용 가능합니다.

블랙박스 모델 + 설명할 데이터 포인트 x를 입력하면, 설명 모델 g가 나온다.

새로운 데이터 z를 샘플링

생성한 z들을 블랙박스 모델 f에 넣어서 예측값 f(z)를 구합니다.

수학적 기초

Fidelity, Complexity, Proximity — LIME의 세 가지 축

최적화 목표

argming∈G L(f, g, φx) + J(g)
• L(f, g, φx) : Local fidelity loss. g가 f를 x 근처에서 얼마나 잘 따라하나
• J(g) : Complexity. g가 얼마나 단순한가
• G : 해석 가능한 모델의 집합 (선형 모델, 트리 등)

실전에서는 J(g)는 사용자가 미리 정하고 (예: "Feature 5개 이하"), L만 최적화합니다.

Local Fidelity (충실도)

L(f, g, φx) = ∑z∈Z φx(z) · [g(z) - f(z)]2
z에서 surrogate model g의 예측과 블랙박스 f의 예측 차이를 제곱하고,
근접도 가중치를 곱해서 합산. x에 가까울수록 더 정확하게 맞춰야 함.

Proximity Kernel (근접도)

φx(z) = exp(-d(x,z)2 / σ2)
• d(x,z) : x와 z 사이의 거리 (Euclidean 또는 Gower)
• σ : kernel width. 이게 "local"의 범위를 결정
• 가까우면 φ ≈ 1, 멀면 φ ≈ 0
Kernel Width σ 의 효과

σ 크면: 넓은 범위를 "local"로 취급. 멀리 있는 점도 영향력 있음 → global에 가까워짐

σ 작으면: 아주 좁은 범위만. 데이터 부족 가능 → noise에 민감

이것이 LIME의 핵심 하이퍼파라미터이자 가장 큰 약점 중 하나입니다.

Complexity 측정

선형 모델의 경우

J(g) = 0이 아닌 계수의 수 (L0 norm)
Feature 5개만 쓰면 J = 5

트리 모델의 경우

J(g) = leaf node의 수
깊이가 깊을수록 복잡

r/datascience
LIME의 트레이드오프를 한 줄로:

Fidelity 높이면 → 모델이 복잡해짐 → 설명이 안 됨
Complexity 줄이면 → 모델이 단순해짐 → 블랙박스를 못 따라함

ML의 bias-variance tradeoff가 여기서도 나오네 ㅋㅋ
이해 확인

LIME의 최적화에서 L(fidelity)과 J(complexity)의 관계는?

트레이드오프입니다. L을 줄이려면(블랙박스를 잘 따라하려면) 복잡한 surrogate가 필요하고, J를 줄이려면(간단한 설명) fidelity를 포기해야 합니다. 실전에서는 J를 먼저 제한(예: L1 정규화로 Feature 5개)하고 L만 최적화합니다.

Lesson 6

LIME의 함정들

LIME은 강력하지만, 조심하지 않으면 틀린 설명을 줄 수 있다.

1. Sampling 문제 (Extrapolation)

Feature 간 상관관계를 무시하고 독립적으로 샘플링합니다.

결과: 현실에 존재하지 않는 데이터로 모델을 평가. 예: 키 190cm + 체중 30kg인 사람.

해결: 훈련 데이터 기반 샘플링, 또는 local sampler 사용

2. Locality 정의 문제 (Kernel Width)

σ를 어떻게 설정하느냐에 따라 설명이 완전히 달라집니다.

자동으로 최적 σ를 찾는 방법이 없습니다.

큰 σ: global에 가까워져서 local 정보 놓침
작은 σ: 데이터 부족, noise fitting

3. Local vs Global Features

전체 입력 공간에서 샘플링하면, 일부 영역에서만 중요한 local feature가 전체적으로 영향력 있는 global feature에 묻힙니다.

비유: 결정 트리에서 root 근처 split은 global, leaf 근처 split은 local. LIME이 local split을 놓칠 수 있음.

4. Faithfulness vs Sparsity 트레이드오프

Feature 3개짜리 선형 모델: 예측 0.283 vs 원래 0.143 (차이 크다!)

Feature 9개짜리 GAM: 예측 0.148 vs 원래 0.143 (거의 같다!)

간단한 설명은 부정확하고, 정확한 설명은 복잡합니다.

5. Bias 은닉 (Adversarial Attack)

악의적 개발자가 in-distribution에서는 편향된 모델을, out-of-distribution(LIME이 생성하는 데이터)에서는 공정한 모델을 사용할 수 있습니다.

결과: LIME은 "공정하다"고 설명하지만 실제로는 편향된 모델!

6. Robustness (불안정성)

매우 가까운 두 점의 설명이 크게 다를 수 있습니다. 랜덤 샘플링에 의존하기 때문입니다.

같은 점을 두 번 설명해도 다른 결과가 나올 수 있음!

r/MachineLearning
LIME pitfall 요약:
- kernel width 바꾸면 설명이 바뀜
- 샘플 바꾸면 설명이 바뀜
- 같은 걸 두 번 돌리면 설명이 바뀜

"reproducible explanation" 이라고 했잖아...
LIME: "나는 그런 말 한 적 없다"
이해 확인

LIME이 bias를 숨기는 데 악용될 수 있는 이유는?

LIME은 설명을 위해 out-of-distribution 데이터를 생성합니다. 악의적 개발자가 in-distribution 데이터에는 편향된 모델을, LIME이 생성하는 OOD 데이터에는 공정한 모델을 사용하면, LIME은 "이 모델은 공정하다"고 보고하지만 실제 사용 시에는 편향된 예측을 합니다. Classifier로 in/out-of-distribution을 구분하는 adversarial attack이 가능합니다.

Lesson 1

"만약에...?" — Counterfactual Explanation

SHAP/LIME은 "왜?"를 답하지만, Counterfactual은 "어떻게 하면?"을 답한다.

The Situation

은행 AI가 당신의 대출을 거절했습니다.

성별: 남 | 나이: 52 | 직업: 비숙련 | 금액: $10,000 | 기간: 24개월 | 용도: TV
→ 결과: 거절

Counterfactual 설명: "직업이 숙련직이고 대출 금액이 $8,000이었다면, 대출이 승인되었을 것입니다."

이것이 counterfactual입니다. 현재 사실과 다른(counter-to-fact) 가상 시나리오를 보여주는 것.

r/ProgrammerHumor
SHAP: "당신의 대출이 거절된 이유는 소득이 -0.3, 나이가 -0.2, 금액이 -0.5..."
사용자: "그래서 내가 뭘 어떻게 하라는 건데?"

Counterfactual: "금액을 8000으로 줄이면 승인됩니다"
사용자: "OK 바로 수정합니다"

actionable insight가 뭔지 보여주는 예시

Counterfactual의 4가지 역할

행동 지침 (Guidance)

"1년 뒤에 더 큰 금액으로 다시 신청하세요"
→ 구체적인 다음 행동을 알려줌

이유 발견 (Reasons)

"나이가 대출 심사에 영향을 주는군요"
→ 몰랐던 요인을 발견

이의 제기 (Contest)

"나이로 차별하다니 부당합니다!"
→ 부당한 결정에 근거 있는 항의 가능

모델 디버깅 (Bias Detection)

"금액을 올렸더니 승인? 이건 버그 아닌가?"
→ 모델의 이상한 행동 발견

철학적 배경 — Lewis (1973)

Counterfactual은 분석 철학에서 오래된 개념입니다.

"S가 사실이었더라면, Q가 사실이었을 것이다."

여기서 핵심: S는 실제로 일어나지 않은 사건이어야 합니다. 그리고 가능한 한 현실과 가장 비슷한 세계에서 Q가 성립해야 합니다.

이 "현실과 가장 비슷한 세계" = counterfactual의 proximity 조건.

r/learnmachinelearning
counterfactual explanation =
if (평행 우주에서_내가_조금만_달랐다면) {
  result = 대출_승인;
}

그 "조금만"이 핵심이고, "얼마나 조금"인지를 수학적으로 최적화하는 게 이 챕터의 전부
이해 확인

Counterfactual과 SHAP/LIME의 가장 큰 차이는?

SHAP/LIME: "왜 이 예측이 나왔는가?" (Feature 기여도 분해)
Counterfactual: "뭘 바꾸면 다른 예측이 나오는가?" (행동 지침)

SHAP/LIME은 과거 지향적(왜 이렇게 됐나), CE는 미래 지향적(뭘 하면 바뀌나)입니다.
Lesson 2

수학적 정의: 최적화 문제

좋은 counterfactual x'을 찾는 것 = 최적화 문제를 푸는 것

Valid Counterfactual의 조건

조건 1: 원하는 예측
f(x') = y' (원하는 결과)
예: "대출 승인"
조건 2: 최소 변경
x'와 x가 최대한 가까워야
"인생을 통째로 바꿔라"는 안 됨

기본 최적화 공식

argminx' λ1 · op(f(x'), y') + λ2 · of(x', x)
• op(f(x'), y') : prediction distance. x'의 예측이 원하는 결과 y'에 얼마나 가까운가
• of(x', x) : feature distance. x'이 원래 x에 얼마나 가까운가
• λ1, λ2 : 두 목표의 가중치 밸런스

Distance 측정

Prediction Distance (op)

회귀: L1 distance = |f(x') - y'|

분류 (score): L1 distance

분류 (label): 0-1 Loss = I{f(x') ≠ y'}

Feature Distance (of) — Gower Distance
dG(x', x) = 1/p ∑j=1p δG(x'j, xj)
수치형: δG = |x'j - xj| / Rj (범위로 정규화)
범주형: δG = I{x'j ≠ xj} (다르면 1, 같으면 0)

Gower distance는 수치형과 범주형 Feature를 동시에 처리할 수 있어서 실전에서 매우 유용합니다.
r/datascience
counterfactual 최적화 = 내비게이션

목적지(y'): "대출 승인" 영역
출발지(x): 현재 나의 상태
최적화 목표: 가장 짧은 경로로 목적지에 도착

근데 이 내비는 "성형수술하고 직업 바꾸세요" 같은
비현실적 경로도 추천함 → 그래서 plausibility가 필요
이해 확인

Gower distance가 Euclidean보다 나은 이유는?

Gower distance는 수치형과 범주형 Feature를 동시에 처리할 수 있습니다. Euclidean은 수치형만 가능합니다. 또한 Gower는 각 Feature를 범위로 정규화하므로 스케일이 다른 Feature 간 공정한 비교가 가능합니다. 대출 데이터처럼 "나이(수치형)"와 "직업(범주형)"이 섞인 경우에 필수적입니다.

of — Proximity (Feature Distance)

"원래와 최대한 비슷해야 한다"

"직업만 바꾸면 됨" vs "직업도 나이도 소득도 전부 바꿔야 됨"

os — Sparsity

"바꾸는 Feature 수가 적어야 한다"

os(x', x) = ∑ I{x'j ≠ xj}

L0 norm = 변경된 Feature의 개수. 사람은 짧은 설명을 선호합니다.

o4 — Plausibility

"현실적으로 가능한 변경이어야 한다"

x'이 훈련 데이터 분포 위에 있어야. "소득 높이고 동시에 실업" 같은 모순 불가.

Plausibility 비유

두 가지 경로로 "대출 승인"에 도달할 수 있습니다.

경로 A: 금액 $10K → $3K (직선, 짧음, 근데 빈 공간을 지남)
경로 B: 금액 $10K → $8K + 기간 24 → 12 (돌아가지만 데이터 밀집 지역 경유)

경로 A는 가깝지만 비현실적(data manifold 이탈). 경로 B는 멀지만 현실적(data manifold 위). Plausibility = 경로 B를 선호하는 것.

r/MachineLearning
Proximity만 최적화하면: "나이를 0.5세만 줄이면 승인!"
→ 실제로는 나이를 줄일 수 없음

Sparsity만 최적화하면: "소득을 10배로 올리면 승인!"
→ Feature 1개만 바꿨지만 비현실적

Plausibility까지 고려하면: "1년 경력 쌓고 금액 줄이면 승인"
→ 드디어 actionable한 조언
이해 확인

Sparsity와 Proximity의 차이는?

Proximity (of): 변경된 Feature 값의 크기가 작아야 함 (Gower distance)
Sparsity (os): 변경된 Feature의 개수가 적어야 함 (L0 norm)

예: Feature 10개를 아주 조금씩 바꾸면 Proximity는 좋지만 Sparsity는 나쁩니다. Feature 1개를 크게 바꾸면 Sparsity는 좋지만 Proximity는 나쁩니다.

생성 방법론

Counterfactual을 실제로 어떻게 찾는가 — 두 가지 접근법

방법 1: Wachter et al. (2018) — 원조

argminx' maxλ λ(f(x') - y')2 + ∑ |x'j - xj| / MADj
• MADj : Feature j의 Median Absolute Deviation (정규화용)
• λ를 점진적으로 키워서, 먼저 prediction을 맞춘 후 distance를 줄임
• 최적화 도구: Nelder-Mead (gradient-free)
한계점

• λ를 사전에 어떻게 설정할지 모름

• 수치형 Feature만 처리 가능

• Sparsity, Plausibility 미고려

• Counterfactual 1개만 반환

방법 2: MOC — Multi-Objective Counterfactual (Dandl et al., 2020)

argminx' (op, of, os, o4)
4가지 목적함수를 동시에 최적화. λ 가중치가 필요 없음!
NSGA-II (multi-objective genetic algorithm) 사용.
Wachter

• 단일 목적함수 (λ로 결합)
• 수치형만
• CE 1개 반환
• 단순하지만 제한적

MOC

• 다목적 최적화 (λ 불필요)
• 수치형 + 범주형
• 다수의 다양한 CE 반환
• 서로 다른 트레이드오프 표현

MOC가 200번 반복 후 찾은 CE: 69개

공통점: 모두 credit duration과 credit amount를 줄이는 것을 제안

일부 CE는 x에서 멀지만, 훈련 데이터 밀집 지역에 위치 (plausibility 우수)

Lesson 5

함정과 한계

Counterfactual도 완벽하지 않다.

1. 모델 이해의 환상 (Illusion of Understanding)

CE를 보면 사용자는 "모델을 이해했다"고 느끼지만, 실제 모델 이해도는 변하지 않습니다.

심리학 연구 결과: 주관적 이해도 (증가) vs 객관적 이해도 (변화 없음)

2. 모델 vs 현실 세계 혼동

CE: "5년 후에 신청하면 승인됩니다" → 5년 뒤 다시 신청했더니 거절?

이유: 5년 동안 다른 Feature(직업, 부채 등)도 변했고, 은행 모델 자체도 바뀜.

CE는 모델의 설명이지 현실의 설명이 아닙니다.

3. Rashomon Effect — 너무 많은 설명

똑같이 좋은 CE가 여러 개 나올 수 있고, 서로 모순될 수도 있습니다.

CE1: "기간을 줄이세요" vs CE2: "기간을 늘리세요" → 사용자 혼란

4. Actionability vs Fairness 충돌

"바꿀 수 있는 것만 바꿔라" → 인종, 성별은 actionable하지 않으니 CE에서 제외

하지만 그러면 인종/성별 차별을 숨기는 결과!

5. 모델 변경 가정 (Constant Model)

CE는 모델이 미래에도 동일하다고 가정. 현실에서는 모델이 주기적으로 업데이트됩니다.

6. 정보 유출 & Adversarial Attack

CE가 모델 정보를 너무 많이 노출. 공격자가 이를 이용해 모델을 역설계하거나, 개발자가 의도적으로 편향된 CE를 생성할 수 있음.

CE pitfall 요약:
- 사용자가 이해한 것 같지만 실은 아님
- 모델 설명인데 현실 조언으로 착각함
- 답이 여러 개인데 서로 모순됨
- 공정성을 위해 만들었는데 차별을 숨김

Trust 높이기

설명을 신뢰할 수 있는지 진단하는 방법 — OOD Detection & Robustness

설명이 신뢰할 수 없는 두 가지 이유

Extrapolation
훈련 데이터가 없는 영역에서
설명을 생성하면 엉터리
Low Robustness
비슷한 입력인데
설명이 크게 다르면 불안

OOD Detection 방법 1: OOD Classifier

훈련 데이터 = in-distribution. 랜덤 생성 데이터 = out-of-distribution.

이 둘을 구분하는 binary classifier를 학습합니다.

CE나 LIME의 샘플이 OOD로 분류되면 → 해당 설명은 신뢰할 수 없음!

OOD Detection 방법 2: DBSCAN Clustering

DBSCAN = Density-Based Spatial Clustering of Applications with Noise

• Core point: ε-neighborhood 안에 m개 이상의 점이 있는 점

• Border point: Core point의 ε-neighborhood 안에 있는 점

• Noise point: 어떤 cluster에도 속하지 않는 점

새로운 CE가 cluster 안이면 in-distribution (신뢰 가능), 밖이면 OOD (주의!)

Robustness 측정 — Lipschitz Continuity

||g(x) - g(x0)|| < ω · ||x - x0||
• g: 설명 함수 (LIME의 계수, SHAP value 등)
• ω: Lipschitz 상수. 0에 가까울수록 robust
• 비슷한 입력(x ≈ x0) → 비슷한 설명(g(x) ≈ g(x0))이어야 함
ω 추정
ω̂(x) = maxx(i) ∈ Nε(x) ||g(x) - g(x(i))||2 / d(x, x(i))
x의 ε-neighborhood에서 설명의 변화율이 가장 큰 값을 찾아 ω를 추정합니다.
r/datascience
LIME robustness 테스트:
- 선형 모델 설명: 비슷한 점 → 비슷한 계수 (robust!)
- Random Forest 설명: 비슷한 점 → 완전히 다른 계수 (not robust!)

"이 설명 믿어도 돼요?" → ω를 먼저 확인하세요
이해 확인

LIME, SHAP, CE 모두에서 OOD가 문제인 이유는?

LIME: surrogate 학습용 데이터가 OOD일 수 있음
SHAP: marginalization에서 생성되는 permuted observation이 OOD
CE: counterfactual x' 자체가 OOD일 수 있음

모든 경우, 훈련 데이터가 없는 영역에서 모델이 예측한 값은 신뢰할 수 없고, 그 값에 기반한 설명도 신뢰할 수 없습니다. OOD detection은 세 방법 모두에 적용되는 공통 진단 도구입니다.

  • 모순되는 CE 가능 (기간 늘리라 vs 줄이라)

시험 답변 핵심 논점

1. CE의 고유한 가치:
SHAP/LIME은 "왜?"에 답하지만, CE는 "어떻게 하면?"에 답함. 사용자 입장에서 가장 actionable한 설명.

2. 4대 목적함수:
Prediction(원하는 결과 도달) + Proximity(최소 변경) + Sparsity(적은 Feature 변경) + Plausibility(현실적 변경). 이 네 가지의 트레이드오프를 이해해야 함.

3. Actionability vs Fairness:
actionable Feature만 바꾸면 → 인종/성별 차별 숨김. 이 딜레마는 시험에서 논술형 답변의 핵심 소재.

4. Trust 진단:
OOD detection(DBSCAN, OOD classifier)과 Robustness(ω 추정)는 LIME, SHAP, CE 모두에 적용되는 공통 프레임워크.

Final Check

CE에서 Actionability만 강조하면 발생하는 Fairness 문제는?

"바꿀 수 있는 Feature만 바꿔라"는 원칙에 따르면, 인종/성별/나이 같은 보호 속성(protected attributes)은 변경 불가로 제외됩니다. 그러면 CE에서 이 Feature들이 빠지고, 모델이 실제로 인종/성별에 기반해 차별적 예측을 하더라도 CE만 보면 차별이 드러나지 않습니다.

해결: Fairness 분석에서는 보호 속성도 CE에 포함시켜서, "인종을 바꾸면 결과가 바뀌는가?"를 명시적으로 테스트해야 합니다.
